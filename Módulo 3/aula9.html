AULA 9 • Módulo 3 – Bancos de Dados

Banco de Dados Profissional (SQLite & MongoDB)

Objetivo da aula: entender por que e quando sair do JSON e migrar para um banco de dados profissional, conhecendo os conceitos básicos de SQLite (relacional) e MongoDB (NoSQL), e como isso se aplica ao bot.

▸ 1. Por que JSON não é suficiente em projetos grandes?

JSON é ótimo para começar, mas tem limitações quando o bot cresce:

  • Arquivos podem ficar grandes demais
  • Díficil lidar com muitas leituras/escritas ao mesmo tempo
  • Maior risco de corromper dados
  • Não foi feito para consultas complexas (filtros, ordenações, etc.)

Quando o bot começa a ter muitos usuários, sistemas de XP complexos, rankings, loja de itens, várias guilds, etc., é hora de pensar em um banco de dados de verdade.

▸ 2. Bancos relacionais x bancos NoSQL

De forma bem resumida, temos dois grandes estilos de banco:

  • Relacional (SQL): tabelas, colunas, linhas, relacionamentos.
  • NoSQL (documentos, chave-valor, etc.): mais flexível, foco em documentos/objetos.

Nesta aula vamos focar em:

  • SQLite: banco relacional leve, salvo em um arquivo.
  • MongoDB: banco de documentos (NoSQL), muito usado em apps web modernas.

▸ 3. SQLite – banco relacional em um arquivo

SQLite é um banco de dados relacional que roda direto em arquivo, sem precisar de servidor separado. Ideal para:

  • Bots médios
  • Projetos pessoais
  • Ambiente de estudo e protótipo

Conceitos importantes:

  • Tabelas: como “planilhas” (ex.: usuarios, servidores)
  • Linhas: cada registro (um usuário, um servidor)
  • Colunas: campos/atributos (xp, nível, moedas, etc.)
Exemplo de tabela de usuários (conceitual):

Tabela: usuarios
Colunas: id_usuario, xp, nivel, coins

Registro 1: (1234567890, 250, 5, 1500)
Registro 2: (9876543210, 80, 2, 270)

Em código (Python/JS), você faz consultas usando SQL, como: SELECT, INSERT, UPDATE, DELETE.

▸ 4. MongoDB – banco de documentos (NoSQL)

MongoDB é um banco de dados orientado a documentos, ótimo para trabalhar com estruturas parecidas com JSON.

Conceitos básicos:

  • Banco: conjunto de coleções (ex.: botsystems)
  • Coleções: “tabelas” de documentos (ex.: usuarios, configs)
  • Documentos: objetos JSON com os dados (cada usuário, cada config)
Exemplo de documento de usuário em MongoDB:

{
  "_id": "1234567890",
  "xp": 250,
  "nivel": 5,
  "coins": 1500,
  "ultima_mensagem": "2025-11-29T20:31:00Z"
}

Em vez de tabelas rígidas, cada documento pode ter campos diferentes, o que dá mais flexibilidade para bots com muitos tipos de dados.

▸ 5. Quando escolher SQLite, quando escolher MongoDB?

SQLite é ideal se:

  • O bot é de médio porte e roda em uma instância só
  • Você quer algo simples de configurar (apenas um arquivo .db)
  • Você gosta do modelo de tabelas e SQL tradicional

MongoDB é ideal se:

  • O bot vai crescer bastante, com muitos dados variados
  • Você quer flexibilidade de documentos (tipo JSON)
  • Pensa em integrar com aplicações web mais complexas

Para muitos bots de comunidade (como o do Systems_BSI), tanto SQLite quanto MongoDB podem funcionar bem — a escolha depende do seu estilo e objetivo de aprendizado.

▸ 6. Modelando dados do bot em SQLite e MongoDB

Vamos imaginar que queremos guardar XP, nível e coins de cada usuário, e também algumas configurações por servidor.

Modelo em SQLite (conceito):

Tabela: usuarios
- id_usuario (PRIMARY KEY)
- xp (INTEGER)
- nivel (INTEGER)
- coins (INTEGER)

Tabela: servidores
- id_servidor (PRIMARY KEY)
- canal_logs (TEXT)
- canal_boas_vindas (TEXT)
- cargo_padrao (TEXT)
Modelo em MongoDB (conceito):

Coleção: usuarios
Documento:
{
  "_id": "id_usuario",
  "xp": 250,
  "nivel": 5,
  "coins": 1500
}

Coleção: servidores
Documento:
{
  "_id": "id_servidor",
  "canal_logs": "1234567890",
  "canal_boas_vindas": "0987654321",
  "cargo_padrao": "1122334455"
}

Perceba que a estrutura é muito parecida com aquilo que fizemos com JSON, só que agora com as vantagens de um banco de dados de verdade.

▸ 7. Resumo da Aula

  • Entendemos por que JSON não é suficiente para bots grandes
  • Vimos a diferença entre bancos relacionais (SQL) e NoSQL
  • Conhecemos o SQLite como opção leve e simples baseada em tabelas
  • Conhecemos o MongoDB como opção flexível baseada em documentos
  • Discutimos quando usar cada um, de acordo com o tamanho e objetivo do bot
  • Modelamos dados do bot em ambos os formatos (usuarios e servidores)

Nas próximas aulas/projetos, esses conceitos podem ser implementados em código real (Python/JavaScript) para que o bot tenha um banco de dados profissional por trás dos sistemas de XP, economia, rankings e muito mais.
